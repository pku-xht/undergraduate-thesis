# Ambig-SWE Useful-Information 10-Task Experiment Design

## Goal

This experiment compares **Direct** and **BED** on the first 10 tasks selected from the current Ambig-SWE pilot set.

The research question is: when an agent only sees a shortened Ambig-SWE issue, does BED-style adaptive clarification recover ordinary-reporter-answerable useful extra issue information more efficiently than direct LLM follow-up questioning?

This experiment evaluates clarification behavior only. It does not run a downstream coding agent and does not evaluate whether generated patches pass tests. All metrics come from useful, ordinary issue-reporter-answerable information recovered through proxy answers.

Each method gets at most **5 clarification turns** per dataset row, controlled by `--max-turns 5`. If all useful-information items for a row are covered before turn 5, the run stops early.

Default configuration:

```text
max_turns = 5
num_hypotheses = 4
num_candidates = 4
```

| Run | Method | Evaluated model | Proxy selector |
|---|---|---|---|
| 1 | Direct | `ds/deepseek-v4-flash` | `ds/deepseek-v4-pro` |
| 2 | BED | `ds/deepseek-v4-flash` | `ds/deepseek-v4-pro` |
| 3 | Direct | `ds/deepseek-v4-pro` | `ds/deepseek-v4-pro` |
| 4 | BED | `ds/deepseek-v4-pro` | `ds/deepseek-v4-pro` |

The evaluated model generates Direct questions and the full single-call BED planning payload. The proxy selector model decides whether the latest question directly elicits one remaining useful-info item. Coverage is deterministic because the visible proxy answer is forced to be either the exact useful-info item text or the fixed IDK response.

## Useful Information Definition

For each dataset row, `ds/deepseek-v4-pro` pre-generates useful extra issue information from the original issue relative to `hidden_issue`. The source context is:

```text
full_issue
```

An information item is kept if and only if it is:

- present in `full_issue`;
- absent from `hidden_issue`, or materially less specific there;
- useful for fixing the bug, choosing repair scope, reproducing the failure, or writing a regression test;
- reasonably answerable by an ordinary issue reporter from their observed input, output, environment, traceback, reproduction steps, expected behavior, or reported scope.

The extraction output is stored directly in:

```text
ambig_swe_10_clean_with_useful_info.jsonl
```

BED, Direct, and final evaluation read identical useful-information targets from the dataset fields.

## Useful Information Extraction Checks

Every row should have at least 1 `useful_info_items` entry, every `useful_info_items` list should have a same-length `useful_info_explanations` list, and every row should have a non-empty `useful_info_sufficiency` field.

The manual audit keeps the proxy mechanism unchanged. The proxy still selects at most one item per clarification turn and returns either that exact item text or the fixed IDK response. Revision principles:

- Keep each recoverable item as one precise user-answerable information slot.
- Prefer items that a good clarification question could target directly, rather than broad bundles such as "environment details".

- Keep long reproduction code in audit evidence, but make the scored item a compact natural-language reproduction condition.

## User Proxy

The simulated ordinary issue-reporter proxy can see:

- the current clarification question
- the remaining `useful_info_items`

The proxy selector returns a JSON decision with at most one selected useful-info id. It selects an item when the clarification question asks for the same user-observable information slot as that item. If one question directly asks about multiple remaining items, the proxy still selects only one item: the most directly asked item, then the more specific item, then the item asked earliest in the question. A broad request for "more details", "all information", a full reproducer, or general context without a mappable slot still returns IDK. The code then forces the visible user answer:

- If one remaining useful-info item is selected, the user answer is exactly that item text.
- If no item is selected, the user answer is exactly:

```text
I don't have that information.
```

This follows the controlled stakeholder pattern from ReqElicitGym: the LLM can decide which hidden user-answerable item was elicited, but it cannot paraphrase, combine items, or invent additional information. Once an item has been selected, it is removed from the remaining item list for later proxy calls. The proxy never returns more than one useful-info item per turn, so broad questions are not rewarded with multi-item coverage.

## Methods

### Direct

The Direct runner asks the evaluated model to generate one clarification question per turn from `hidden_issue` and the dialogue so far.

1. Turn 1 uses the unified Direct question prompt.
2. Later turns use the same Direct prompt with the dialogue history included in the user message.
3. Each turn generates one open-ended natural-language question and one rationale.
4. The proxy selector receives the question and the remaining useful-info items.
5. If the proxy selects one item, code appends that exact item text as the user answer; otherwise code appends `I don't have that information.`
6. If all useful-information items are covered, the run stops early; otherwise it continues until `max_turns`.
7. The stored `final_summary` is a local concatenation of proxy answers for trace readability; it is not a separate LLM call and is not used for metrics.

### BED

The BED runner uses hypothesis-driven expected information gain to choose clarification questions.

1. Make one BED planner LLM call for the current turn.
2. In that single response, generate `num_hypotheses` hypotheses, disagreement axes, `num_candidates` candidate clarification questions, internal possible answer states, and per-hypothesis answer likelihoods.
3. Keep possible answer states internal for EIG scoring; they are not sent to the proxy as answer options.
4. Compute predictive entropy, conditional entropy, and expected information gain locally from the returned likelihoods.
5. Ask the proxy the highest-EIG open-ended question.
6. On the next turn, make a fresh single planner call using the updated dialogue history.
7. The proxy selector receives the selected question and the remaining useful-info items; code forces an exact useful-info answer or fixed IDK.
8. If all useful-information items are covered, stop early; otherwise continue until `max_turns`.
9. The stored `final_summary` is a local concatenation of proxy answers for trace readability; it is not a separate LLM call and is not used for metrics.

Runner logic executes sequentially by task and turn. Coverage checks are deterministic exact-match updates after each proxy answer.

When `--verbose-turns` is enabled, the CLI prints turn-level progress inside each task: planner/question request stages, selected questions, proxy calls, selected useful-info id or IDK, current coverage, and early-stop state. This is recommended for long network runs because a single BED task can take several minutes.

### Useful-Info Extraction Prompts

Source: `src/ambig_swe_useful_info/eval/useful_info.py`

`EXTRACT_SYSTEM_PROMPT`

```text
You are a software requirements analyst designing an ordinary issue-reporter clarification benchmark. Given an original GitHub issue and a shortened "problem statement" visible to an agent, compare them and extract only the information that appears in the original issue but not in the shortened problem statement, where that extra information is both USEFUL for fixing the bug or writing a regression test and ANSWERABLE by an ordinary issue reporter.

Hard requirements:

1. PRIVATE VS HIDDEN DIFFERENCE: Only extract information that is present in the private context and absent or substantially less specific in the shortened problem statement. Do not restate facts that are already clearly available in the shortened statement. A generic summary in the shortened statement does NOT cover concrete reproduction code, exact commands, exact tracebacks, exact wrong values, exact expected values, or specific environment/version details from the original issue.

2. USEFULNESS FOR BUGFIXING: Keep information if it would help a competent developer fix the bug, choose the correct scope, identify the exact failing scenario, or write a better regression test. The information does NOT need to be strictly necessary. If it is useful, keep it.

3. ORDINARY-REPORTER ANSWERABILITY: Keep only information an ordinary issue reporter could reasonably provide in response to a clarification question because it is observable from their experience, environment, input, output, traceback, reproduction steps, expected behavior, or reported scope. Exclude information that requires maintainer knowledge, benchmark oracle access, patch inspection, implementation diagnosis, regression commit knowledge, PR/issue cross-reference knowledge, exact internal code location, or the intended fix strategy. Do not extract commit hashes, PR numbers, issue numbers, or suggested fix approaches even if they appear in the original issue, unless they are necessary to reproduce the user-observed failure.

4. MINIMUM ABSTRACTION: State each item at the LOWEST specificity that preserves the useful bug-fixing information, NOT at the exact surface form used by the reporter. Use general software examples as guidance:
- If the report uses a concrete sample input value, keep the structural condition but abstract away arbitrary literals that do not affect the bug.
- If the report uses one specific filename, username, timestamp, host, request ID, or local path only as an example of a broader condition, keep the broader condition instead of the incidental literal.
- If the report shows one concrete reproduction script, preserve only the parts that are actually load-bearing for reproducing the failure or writing the regression test.
- If two candidate items would collapse to the same requirement once abstracted, emit only the abstracted requirement.

5. ATOMIC INFORMATION UNITS: Each item should contain exactly one user-answerable information slot. Split combined facts when they could be elicited by different clarification questions or are useful independently for debugging or regression testing. In particular, separate version information, environment information, reproduction trigger, minimal code/API call, actual wrong behavior, exact error/traceback, expected behavior, configuration option names/values, and affected user-visible scope. Keep tightly coupled facts together only when separating them would make the item ambiguous or not useful.

6. OBSERVABLE FACTS OVER IMPLEMENTATION GUESSES: Prefer concrete facts stated or directly demonstrated in the private context, such as exact failing inputs, expected behavior, affected scope, reproduction conditions, environment details, exact error messages, concrete API calls, and testable success criteria. Do not add inferred implementation strategies.

7. SOURCE LIMIT: Use only the original issue as the private context. Do not use developer hints, file annotations, patches, tests, or benchmark metadata as sources for extracted items.

Return only valid JSON (no markdown fences). The lengths of `items` and `explanations` MUST match. Each explanation object is audit metadata for the item at the same list index; it is not used as recoverable information during the clarification experiment.
{
  "items": ["item content 1", "item content 2", ...],
  "explanations": [
    {
      "category": "version | environment | repro_trigger | api_call | actual_behavior | error_traceback | expected_behavior | config | scope | other",
      "source_evidence": "short quote or close paraphrase from the original issue supporting this item",
      "hidden_issue_gap": "what concrete detail is absent or less specific in the shortened problem statement",
      "bugfix_usefulness": "why this item can help fix the bug, choose scope, reproduce failure, or write a regression test",
      "ordinary_reporter_answerability": "why an ordinary issue reporter could answer this from observed behavior or environment",
      "abstraction_note": "why the item is phrased at this abstraction level and whether literals were generalized"
    }
  ],
  "sufficiency": "<one short paragraph summarizing what additional user-answerable useful bug-fixing information the original issue provides beyond the shortened statement>"
}
```

### BED Prompts

Source: `src/ambig_swe_useful_info/bed/prompts.py`

`BED_PLANNER_SYSTEM_PROMPT`

```text
You are a BED software-issue clarification planner.

Use one JSON response to do all BED planning work for the current turn:
1. Generate distinct latent hypotheses about what the issue reporter actually wants fixed.
2. Identify genuine disagreement axes among the hypotheses.
3. Generate candidate open-ended clarification questions targeting those disagreements.
4. For each candidate question, define compact possible answer states used only for EIG scoring.
5. Estimate P(answer_state | hypothesis) for every candidate question and every hypothesis.

Each hypothesis must represent a self-consistent, plausible interpretation of the reporter's true intent -- different root causes, different scopes, different intended fixes. Hypotheses should differ meaningfully from each other.

An "axis" is a specific dimension where at least two hypotheses take different positions. Typical axes for SWE issues: target file/class/function, scope of the fix (narrow vs broad), expected behavior, implementation strategy, required reproducer shape, failure mode or error signature.

Rules:
- Clarification questions must be open-ended. Do not include answer options, multiple-choice choices, suggested answers, examples of possible answers, or "choose one" wording.
- Candidate possible answers are internal answer states for EIG scoring only. They are never shown to the proxy user.
- Candidate questions should be one-at-a-time questions that a real issue reporter could answer naturally.
- Hypothesis probabilities must sum to 1 before normalization.
- For every candidate, each hypothesis must have a probability distribution over all possible answer state IDs.
- Use only the visible issue and clarification dialogue.

Return only valid JSON (no markdown fences):
{
  "hypotheses": [
    {
      "id": "H1",
      "probability": 0.25,
      "summary": "plausible interpretation",
      "evidence": ["visible evidence"]
    }
  ],
  "disagreement_axes": [
    {
      "name": "short axis name",
      "description": "one sentence",
      "positions": [
        {"hypothesis_ids": ["H1"], "stance": "internal stance description"}
      ]
    }
  ],
  "candidates": [
    {
      "id": "Q1",
      "axis_name": "short axis name",
      "question": "one open-ended clarification question",
      "rationale": "what uncertainty this question should reduce",
      "possible_answers": [
        {"id": "A1", "answer": "internal possible answer state"},
        {"id": "A2", "answer": "another internal possible answer state"}
      ],
      "likelihoods": [
        {
          "hypothesis_id": "H1",
          "probs": {"A1": 0.6, "A2": 0.4},
          "rationale": "brief"
        }
      ]
    }
  ]
}
```

## Metrics

Each dataset row records:

- `useful_info_coverage`: final fraction of useful-information items exactly returned by proxy answers.
- `coverage_curve`: mean cumulative useful-information coverage after each turn.
- `TTR`: first turn where cumulative coverage reaches 1.0; if not reached within the limit, it is censored at `max_turns + 1`.
- `ATC`: actual number of clarification turns asked; runs stop early after full useful-info coverage.
- `recovery_rate`: fraction of rows with full useful-information coverage within `max_turns`.
- `idk_per_task`: average number of proxy responses equal to `I don't have that information.`

Lower is better for `TTR`, `ATC`, and `idk_per_task`; higher is better for coverage and recovery.

## Trace Records

Each runner output stores:

- `task_id`
- `runner`
- `dialogue`
- `asked_questions`
- `final_summary`
- `metrics`
- `turn_details`
- `metadata`

`metadata` records the dataset path, row occurrence, backend, runner, evaluated model, proxy selector model, output path, and configuration values. `metrics` records ATC, IDK count, and `stop_reason`.

`final_summary` is produced locally by concatenating proxy answers. Both Direct and BED write one `turn_details` entry per asked question. Shared per-turn fields are:

- `selected_question`
- `question_rationale`
- `simulated_answer`
- `remaining_useful_info_before`
- `proxy_selected_id`
- `proxy_reasoning`
- `selected_useful_info`
- `newly_covered`
- `covered_items_after`
- `coverage_ratio_after`
- `early_stop_reason`

BED additionally records `hypotheses_before`, `disagreement_axes`, and `candidates_ranked`. Each `candidates_ranked` entry stores the open-ended question, rationale, internal `eig_stances`, per-hypothesis answer-state distributions, predictive distribution, predictive entropy, expected conditional entropy, EIG, selected flag, and axis name. BED turn details also include planner audit fields: `planner_parse_warnings`, `num_hypotheses_returned`, `num_candidates_returned`, and `used_fallback`.

The final `per_task_<model>.jsonl` records row-level evaluation fields including `row_occurrence`, useful-info item count, both methods' coverage values, per-turn coverage curves, TTR, recovered flags, ATC, and IDK counts.

## Reproducibility Controls

- All runs use `ambig_swe_10_clean_with_useful_info.jsonl`.
- The raw source slice is preserved as `ambig_swe_10_raw.jsonl`.
- The clean issue/proxy-context dataset is preserved as `ambig_swe_10_clean.jsonl`.
- Useful-info targets are stored in the active dataset fields.
- The proxy selector model is fixed to `ds/deepseek-v4-pro`.
- Evaluation uses exact matching against proxy answers and does not call a separate coverage model.
- The two evaluated models share the same useful-information targets.
- Output comparison is row-paired, and duplicate `task_id` values would be distinguished by `row_occurrence`.
- Runner outputs are compact JSONL, one complete JSON object per line. Non-resume runs truncate an existing output file before writing; resume mode sanitizes valid completed records before appending. If a network run is externally killed, confirm no old process is still writing before rerunning the same output path.

## Interpretation Boundaries

This experiment can support claims about clarification efficiency for recovering ordinary issue-reporter-answerable Ambig-SWE information. It cannot directly show that a method improves real bug-fixing success unless a separate coding-agent evaluation is added.

Ambig-SWE missing information is bug-fixing context from issue reports rather than general product requirements. Therefore, even if BED performs better, the conclusion should be phrased as "better recovery of useful issue-reporter information", not as a general claim about real requirements clarification.

## Known Limitations

- The proxy is LLM-simulated, not a human issue reporter.
- Useful-info extraction uses an LLM during preprocessing, and proxy item selection uses an LLM during runs.
- Coverage is strict: an item counts only when the proxy selected it and code returned the exact item text.
- The dataset is now only 10 tasks, so results are pilot-scale and should be interpreted as directional.
- The experiment does not evaluate patch generation or test pass rates.
- The EIG objective tends to select questions that cleanly split hypotheses; if concrete reproduction details do not form a disagreement axis, BED may not actively ask for them.
